# Sails Protocol — RFCs

Índice de leitura sugerido — cada RFC segue o mesmo formato: Summary,
Motivation, Alternatives Considered (incluindo o que foi rejeitado e por
quê), Decision, Primitives Used or Extended, Principle Alignment,
Specification, Backward Compatibility, Reference Implementation Plan.

**🔷 = Core RFC** (`GOVERNANCE.md` §6A) — muda o *wiring* do lifecycle
de um primitive já existente, ou o modelo de confiança/custódia de algo
que `PROTOCOL_INVARIANTS.md` trata como invariante constitucional.
Classificação, não renumeração — ainda entra na mesma sequência única e
permanente (§5). Ganha `Motivation`/`Decision`/`Implementation Impact`
como seções obrigatórias e explícitas, além do template padrão.

| RFC | Título | Resolve |
|---|---|---|
| RFC-001 | Participant Model | Core depende de `Participant` abstrato, não de `Identity` diretamente. Corrige explicitamente por que `Wallet` e `Agent` NÃO são implementações diretas |
| RFC-002 | Transport Provider Abstraction | Pears/HyperDHT vira implementação de `TransportProvider`, não dependência fixa — decisão explícita entre tratar Pears como fundacional (rejeitado) ou criar a interface (aceito) |
| RFC-003 | Proof Primitive | `confirmFiat()` → `submitProof()` genérico. `claimType` fica string aberta, nunca enum fechado |
| RFC-004 | Negotiation State Machine | Chat deixa de ser a primitive; `NegotiationEvent` estruturado passa a ser a abstração, chat vira uma implementação de canal entre outras |
| RFC-005 | Capability Model | Resolve uma colisão de nomenclatura real: "Capability" tinha dois significados nunca diferenciados (categoria funcional de módulo vs. permissão de acesso) — agora são duas interfaces: `Capability` e `CapabilityGrant` |
| RFC-006 | OpenProof Module & Packages | `Proof` ganha módulo dono (8º módulo oficial). `Package` vira a palavra para agrupar vários módulos numa entrega de negócio (ex: "OpenP2P Package") — deliberadamente diferente de `Capability` (RFC-005), que continua 1:1 com um módulo só |
| RFC-007 | Real-World P2P Requirements | Consolida 11 requisitos de operadores P2P (Proof Registry, `EvidenceProvider`, `PendingBankSettlement`, escalonamento de disputa + `ArbitrationProvider`, Timeline, Evidence Bundle, Social Engineering Agent, reputação baseada em Outcome, Operational Profiles) estendendo módulos existentes — sem módulo novo. `EvidenceBundle` e `Timeline` foram avaliados contra o teste de primitive (§1.10-1.11) e rejeitados como primitives |
| RFC-008 | Verifiable Timestamps and a Hash-Chained Timeline | Fecha duas lacunas do modelo inspirado em Nostr que RFC-007 introduziu: `EvidenceReference.timestamp` era auto-declarado (não prova existência-no-tempo, só assinatura), e `TimelineEntry` era uma projeção plana sem vínculo criptográfico entre entradas (editável sem detecção). Adiciona `TimestampAnchor` (novo Adapter, policy-gated, não obrigatório) e encadeia `TimelineEntry` por hash (`entryHash`/`prevHash`, persistido em `EscrowEvent`/`ReputationEvent`) — sem blockchain, sem novo primitive |
| RFC-009 | Decimal Precision for Financial Fields | Auditoria externa (CISO/Chief Architect) apontou `priceUsd`/`amount`/`lockedAmount`/etc. como `Float` (IEEE 754) no schema real — verificado independentemente antes de aceitar a alegação. Muda essas colunas para `Decimal` e propaga a correção como string decimal (nunca `number`) em todo contrato TypeScript cross-module, para que a precisão não se perca de novo ao cruzar um limite JSON. Corrige o tipo do campo `Settlement.amount` (§1.5) — não é uma nova primitive |
| RFC-010 | Durable Event Store and Mandatory correlationId | Mesma auditoria (Event Bus em `EventEmitter` puro, sem durabilidade) refinada por uma segunda revisão externa (CTO) que corrigiu a recomendação original — "eventos precisam ser duráveis" é requisito de protocolo, "use Redis" não é. Introduz `EventStore`, novo Adapter (mesma categoria de `SettlementProvider`), e exige `correlationId` obrigatório em todo evento (`tradeId` hoje, `intentId` quando Intent existir). `InMemoryEventStore` funcional e verificado; `RedisStreamsEventStore` desenhado mas lança erro — não testado contra Redis real, mesmo padrão de `LightningHodlProvider` |
| RFC-011 | P2P Reconciliation on Peer Reconnect | Terceiro e último achado da mesma auditoria: mensagem `HyperDHT`/Pears pode cair enquanto Postgres já registrou o `TradeIntent`. Verificado que a arquitetura já é híbrida (Postgres autoritativo + Pears como canal de notificação em tempo real) — `sendEvent()` já persiste toda mensagem independente de entrega P2P, faltava só o catch-up. `peer.connected` (handshake real, com `localUserId`) agora dispara `ReconciliationService` contra Postgres, sem precisar de replay peer-to-peer. Achado relacionado sinalizado, não corrigido: `NegotiationService`'s status é `Map` em memória, não persistido |
| RFC-012 | Intent Validation and Coordination States | Disparado por uma necessidade concreta: `BuyerAgent` (QVAC, geração autônoma de Intent) precisava de um caminho formal e auditável para o Core receber e processar um Intent antes da Discovery. Verificado contra `PROTOCOL_SPECIFICATION.md` §2.4 antes de escrever — `VALIDATED`/`COORDINATED` não existiam na spec congelada, então é mudança real de Core primitive, não wiring. Adiciona os dois estados entre `CREATED` e `DISCOVERING`, ambos formalizando lógica que já rodava sem ser observável (CISO Byzantine/Economic Rules para `VALIDATED`; primeira implementação real de `coordinationEngine.decide()`, antes stub, para `COORDINATED`). Também consolida `IntentStatus`, que estava declarado duas vezes de forma independente (`common/types/intent.ts` e `core/state-machine.ts`) |
| RFC-013 | Capability Registry Implementation, WalletAdapter Pattern, and Portable Identity via peerId | Disparado por uma proposta técnica externa trazida pelo dono do projeto ("Sails Trading Protocol SDK — Documento Técnico Consolidado v1"). Verificado contra a documentação oficial (docs.pears.com, docs.qvac.tether.io) e o código já real antes de implementar — a proposta atribuía "reputação" a Pears e "verificação de capacidades/políticas" a QVAC, nenhum dos dois é real (Pears é rede P2P; QVAC é inferência de IA local). Ambas as atribuições corrigidas: identidade portátil via `peerId` (Pears) + reputação computada pelo Sails OpenReputation; verificação de capacidade é o próprio Capability Registry (RFC-005), stubado desde o Architecture Freeze — implementado de verdade agora. Também adiciona `WalletAdapter` ao `@sails/sdk` e estende `TradeIntentPayload` com `minReputationRating`/`kycRequired` (aditivo, sem novo tipo de Intent) |
| RFC-014 | Capability Registry Enforcement | RFC-013 deixou `capabilityRegistry` real, mas sem nenhum chamador real no caminho que move dinheiro — achado explícito de uma auditoria de rigor pedida pelo dono do projeto antes de revisão técnica externa (Tether). Adiciona os dois pontos reais de enforcement: `intentEngine.create()` (antes de persistir um `TradeIntent`) e `executeSettlement()` (antes da liberação real de USDT assinada), ambos atrás de `config.features.enforceCapabilities` (padrão `false`, mesmo precedente de `AUTO_SETTLE_ON_MATCH`) — para não quebrar todo deploy/teste existente, que hoje não tem nenhum `CapabilityGrant` emitido |
| RFC-015 | Two-Person Control for Escrow Release | Custódia neste projeto é single-seed (`WDK_SEED_PHRASE` controla treasury E toda sub-conta de escrow) — achado explícito da mesma auditoria de rigor, com instrução direta do dono do projeto para deixar isso claro "principalmente na prática quando acontece uma transação p2p entre duas partes". Real multisig on-chain foi investigado e rejeitado para este passo (`@tetherto/wdk-wallet-evm-erc-4337` é single-owner, verificado contra os types compilados reais do pacote). Em vez disso, adiciona um controle maker-checker de camada de aplicação: `escrow.service.ts`'s `releaseFunds()` — o único ponto real por onde toda liberação passa — agora exige aprovação de ambas as contrapartes do trade (`Trade.buyerId` e `sellerId`) antes de liberar, no caminho normal (não-disputado); liberações arbitradas (`DISPUTED`) sempre ignoram essa checagem. Atrás de `config.features.requireDualApprovalForRelease` (padrão `false`). Também corrige um furo real encontrado na própria RFC-014: sua checagem de capability vivia só em `settlement-orchestrator.ts`, deixando passível a rota direta de release e a liberação arbitrada — ambas as checagens (RFC-014 e RFC-015) agora vivem dentro de `releaseFunds()` |
| RFC-016 | QVAC as a Crypto-Native Agent — the Fiat/Crypto Boundary | Instrução direta do dono do projeto: "O QVAC nunca deve tocar em PIX. Essa decisão precisa ser arquitetural, não apenas uma escolha da Satsails." Formaliza uma invariante que já era verdadeira por construção (nenhum código deste repositório jamais chamou uma API bancária) mas nunca havia sido declarada como regra: QVAC/`BuyerAgent`/`SellerAgent` só agem sobre ativos digitais já na wallet do usuário, via WDK — nunca sobre trilhos fiat. A conversão PIX→cripto é responsabilidade de um provedor regulado (ex: Moon, Banxa, Topper, Bipa), fora do escopo do Sails Protocol e do QVAC, formalizada no diagrama PIX → Provedor Regulado → USDT/DePix/USDC → Wallet Não Custodial → QVAC Agent → Sails Protocol. Fixa a terminologia "Crypto-Native Agent" (nunca "PIX Agent"/"Banking Agent") e esclarece via doc-comment que `requestUsdtViaPix()` descreve o método de pagamento esperado da contraparte humana, não uma ação que o agente executa contra um banco. Não muda comportamento de runtime — reforça a "canonical fiat model statement" (v8.0) já congelada em `PROJECT_CONTEXT.md`, sem reescrevê-la |
| RFC-017 | Real Timeline Read-Model and Social Engineering Agent | Implementa de verdade RFC-007 D5 (Timeline) e D7 (Social Engineering Agent), ambos especificados mas nunca construídos ("🔲 Not started" em `BACKLOG.md`). Achado real antes de implementar: D5 nomeia seu id `intentId`, mas nenhum evento que um detector de golpes precisaria ver (mensagens de chat, mudanças de status de escrow) usa `intentId` como correlationId hoje — só os eventos do próprio `intent-engine.ts` usam. Corrigido: `Timeline` fica indexado por `correlationId` genérico (`tradeId` na prática hoje), sem mudança de código quando Intent-para-Trade existir. `EventStore` ganha `getEvents()` (real no `InMemoryEventStore`, lança erro no `RedisStreamsEventStore`, mesmo padrão de `publish`/`subscribe`). `SocialEngineeringAgent.evaluate()` (interface do D7, inalterada) usa QVAC para detectar `off_channel_migration`/`payment_instruction_change` a partir de mensagens de chat reais — `unexpected_flow_deviation` fica fora desta passada (precisaria de consciência de máquina de estados, escopo maior, sinalizado explicitamente). Detecção liga um novo evento (`agents.social_engineering.risk_detected`) que vira `RISK_WARNING` no WS do chat — nunca bloqueia ou altera o trade. Atrás de `config.features.socialEngineeringDetection` (padrão `false` — chamada real ao QVAC por mensagem tem custo). 13 novos testes, suíte completa (206 testes) verificada verde |
| RFC-018 🔷 | Intent as the Canonical Entry Point for Every Trade | Auditoria de fidelidade (spec vs. código real, 2026-07-19) achou que `Intent`/`IntentEvent` e `Offer`/`Trade` são hoje dois caminhos de código totalmente desconectados — nenhum `Offer`/`Trade` real tem uma linha de `Intent` por trás, apesar de `PROTOCOL_SPECIFICATION.md` §1.11 já afirmar o contrário conceitualmente. Registra a arquitetura-alvo (`Offer`/`Trade` ganham `intentId` FK; `createOffer()` chama `intentEngine.create()`; `createTrade()` dispara `COMMITTED`; settlement completo dispara `FULFILLED`) e um plano de migração incremental em 3 fases — não implementa a correção nesta passada, por instrução direta do dono do projeto ("apenas registrar... e planejar a migração de forma incremental") |
| RFC-019 🔷 | Settlement Custody — Reference Implementation vs. Normative Protocol | Achado mais grave da mesma auditoria: o único `SettlementProvider` real (`WdkSettlementProvider`) assina toda liberação de escrow com uma única seed do servidor — nenhuma assinatura do usuário é exigida. Isso viola `PROTOCOL_INVARIANTS.md` Invariante Constitucional 2 ("o protocolo nunca custodia ativos") na prática, não só na documentação. Reclassifica essa implementação explicitamente como "Reference Implementation" para desenvolvimento local/testnet, nunca o modelo normativo do protocolo, e registra a arquitetura-alvo (a wallet do usuário assina localmente, o servidor nunca segura uma chave capaz de mover fundos em escrow) mais um plano de migração incremental (Fase 1: flag de custódia + aviso no boot + ponteiros de doc, de baixo risco; Fase 2: settlement realmente não-custodial, engenharia real, sem prazo comprometido). Bloqueante para qualquer uso em produção com valor real até a Fase 2 existir |

Todas as 6 RFCs de RFC-001 a RFC-006 nasceram do mesmo processo: achados
da Protocol Quality Review, revisados e, em alguns casos, corrigidos antes
de virarem decisão final — cada RFC documenta explicitamente o que foi
considerado e rejeitado, não só o que foi aceito. `RFC-007` nasceu de
entrevistas com operadores de mercado, não da Protocol Quality Review, e
foi aceito por diretiva direta do CTO em vez de passar pela janela de
Discussion aberta (`GOVERNANCE.md` §5) — registrado explicitamente no
próprio RFC, não omitido. `RFC-008` emenda duas construções que o próprio
RFC-007 introduziu (`EvidenceReference`, `TimelineEntry`); nasceu de uma
crítica de design levantada durante a revisão do próprio RFC-007, não de
diretiva do CTO — provenance registrada com precisão no próprio RFC.
